Breaking the Compilation Boundary: Semantic Fuzzing of the SBCL Runtime

מאת: שי מרדכי | קטגוריה: Vulnerability Research, Compiler Internals
סטטוס: Architectural Behavioral Research — התנהגויות אלו מעוגנות במפרט הרשמי של השפה.
תקציר המחקר: מ-Zero Day ל-Trust Model
במהלך מחקר Fuzzing סמנטי על קומפיילר SBCL (שפת LISP), זיהיתי שלוש דרכים שונות להפר גבולות לקסיקליים בזמן קומפילציה. בתחילה, לצד קומיטים מחשידים בקוד המקור, הממצאים נראו כמו "Zero-Day" קלאסי של עקיפת בידוד שזכה לתיקון שקט (Silent Patch). אולם, יישום מתודולוגיית "Trust but Verify" ובדיקות דיפרנציאליות הפריכו את ההשערה הראשונית שלי. המסקנה שאליה הגעתי מרתקת הרבה יותר מבאג נקודתי: מדובר בכשל של גבול אבטחתי (Compiler Trust Boundary Failure) המובנה ארכיטקטונית בשפה, אשר פותח דלת למודל תקיפה של Supply Chain Poisoning.

פרק א': ההשערה והווקטורים — אשליית ה-CVE

המחקר החל בשאלה: כיצד הקומפיילר מתמודד עם גבולות הבידוד של משתנים לקסיקליים בזמן פריסת מאקרו (Macro Expansion)? להבדיל משפות קשיחות כמו Java האוכפות בידוד לקסיקלי נוקשה, בניתי Fuzzer סמנטי שהצליח לבודד שלושה וקטורים אבטחתיים ששברו את הבידוד ב-SBCL:

1. הזרקת קישור דינמי (Dynamic Binding Injection)

ניצול הפקודה proclaim בתוך המאקרו כדי לדרוס את המטא-דאטה הגלובלי של הסמל, וכפייה על הקומפיילר לייצר קריאה דינמית (Dynamic Runtime Lookup) במקום הפניה למחסנית (Lexical Stack Offset).

;; PoC Code: isolation-bypass-attack
(defmacro isolation-bypass-attack (target new-value)
  `(progn
     (proclaim '(special ,target)) ; Injecting dynamic binding at compile-time
     (setf ,target ,new-value)))   ; Overriding the lexical value

(defun verify-vulnerability ()
  (let ((secret-data "PROTECTED_DATA"))
    (isolation-bypass-attack secret-data "COMPROMISED")
    (if (string= secret-data "COMPROMISED")
        (format t "[!] VULNERABILITY CONFIRMED~%"))))

2. זיהום טבלת הסמלים (Global Symbol Table Pollution)

שימוש ב-intern כדי להשיג מצביע גלובלי למשתנה המקומי המוגן, ושתילת נתונים על גבי ה-Property List שלו (plist), מה שמאפשר חילוץ מידע לסביבה החיצונית.

3. בלבול מרחבי שמות (Cross-Package Namespace Confusion)

הרצת פקודת shadow מתוך המאקרו שוברת את טבלת המעקב הלקסיקלית ומפנה את המשתנה ממרחב מוגן ל-Namespace ציבורי.

פרק ב': מבחן המציאות — Trust but Verify

בשלב זה, נראה היה שיש בידי פגיעות קלאסית. החשד גבר כאשר זיהיתי קומיטים היסטוריים ודחופים בקוד המקור של SBCL (באפריל 2026), כגון הוספת מנגנוני pseudo-atomic סביב פעולות ה-dynbind ב-cell.lisp ושינויים ארכיטקטוניים ב-TLS. זה נראה כמו Silent Patch מושלם.

אך חוקר אבטחה נמדד ביכולתו להפריך את עצמו.
ביצעתי הרצה דיפרנציאלית (Differential Testing): הרצתי את ה-PoCs על גרסת ה-Baseline הישנה (1.4.3) ועל הגרסה החדשה וה"מתוקנת" ביותר. התוצאה הייתה חד-משמעית: הפגיעות עבדה בשתי הגרסאות ללא דופי.

המסקנה (The Pivot): ההתנהגות שמצאתי אינה באג תוכנה או פגיעות קוד, אלא התנהגות ארכיטקטונית By Design המעוגנת במפרט ה-ANSI. בשפת Lisp, מאקרואים מריצים קוד בזמן קומפילציה (Code is Data), ולכן מותר להם חוקית לעשות מניפולציות עם proclaim או intern. הפאצ'ים המחשידים ב-cell.lisp נועדו לפתור בעיות סנכרון תהליכונים (Thread Safety) ו-GC, ולא נועדו ליצור Sandbox אבטחתי שאינו קיים בשפה.

פרק ג': לסיכום — Compiler Trust Boundary Failure

העובדה שזהו אינו "Zero-Day קלאסי" אינה מורידה מהערך האבטחתי של הממצא, אלא ממקדת אותו בקטגוריה של Compile-Time Trusted Code Execution Model. התגלית האמיתית פה היא ש-Common Lisp לא מספקת גבול אבטחתי (Trust Boundary) בין קוד הפרויקט לבין קוד המאקרו שרץ בזמן קומפילציה.

Supply Chain Poisoning:
בסביבות CI/CD מודרניות, מפתח שמייבא ספריית צד-שלישי תמימה-למראה עלול להריץ "מאקרו מורעל". המאקרו הזה יכול לנצל את הלוגיקה החוקית שמצאנו כדי לעקוף גבולות לקסיקליים, לחלץ מפתחות או סודות, ולזהם את הסביבה — הכול בזמן ה-Build, מבלי להותיר שום זכר בקוד הבינארי הסופי שיזוהה על ידי כלי DAST או RASP. במערכות שבהן הקומפיילר מריץ קוד, ההנחה שלקומפיילר יש גבולות אבטחה היא ההונאה הגדולה ביותר.
