תקציר מנהלים [cite: 5]

הרבה מאוד משתמשי אנדרואיד (ואפילו מפתחים) לא מודעים לנתון הבא: [cite: 6] "עולם האנדרואיד" כפי שאנחנו מכירים אותו, לא באמת קיים ברגע שהמכשיר נדלק. [cite: 7] בשניות הראשונות, הטלפון שלכם פשוט פלטפורמת Linux רגילה לחלוטין. [cite: 8]

המאמר מתמקד בשבריר השניה לאחר ריצת האפליקציה על מכשיר האנדרואיד, וכיצד נוזקות מתקדמות מנסות "לעצור את הזמן" ולהידמות למערכת ההפעלה בעצמה, עוד לפני שהמשתמש רואה מסך כלשהו. [cite: 9]

במאמר זה, ננתח חלק מהנוזקה Anatsa - כלי תקיפה ממשפחת ה-Banking Trojans, המיועד לגניבת פרטי גישה לחשבונות פיננסיים. [cite: 10] ב-2024 היא הגיעה ל-100,000 הורדות ב-Google Play עד שהסירו אותה. [cite: 11] הנוזקה הזו, מבצעת עקיפה לא שיגרתית של הצגת ה-UI של האפליקציה. [cite: 12]

בחרתי לנתח אותה, כמקרה בוחן ל-Framework שפיתחתי למידול התנהגות נוזקה לפי 5 שכבות, ו-Anatsa פשוט ביצעה כל אחת מהן. [cite: 13]

המירוץ מתחיל [cite: 50]

כאשר משתמש מריץ אפליקציה לגיטימית, עץ התהליכים נראה כך: [cite: 51]

ה-Zygote הוא בעצם טמפלט נקי עבור כל אפליקציית אנדרואיד. כדי להכין את הקרקע, תהליך ה-Zygote מאתחל שלושה רכיבים בסיסיים: [cite: 62]

  • ה-Android Runtime: ה-ART הינו "המכונה הוירטואלית", המסוגלת לקרוא ולהריץ קוד Java או Kotlin [cite: 65] (קבצי ה-DEX). [cite: 66]
  • ה-Android Framework: טעינה מראש של אלפי מחלקות ומשאבי מערכת אל הזיכרון. [cite: 67]
  • סוקט מקומי: ערוץ תקשורת שמאזין לבקשות fork(). [cite: 68] יוצר אותו לפני שעדיין ה-System Server נולד. [cite: 68, 69]

הפיצול [cite: 70]

במהלך ה-Boot, מתבצע fork() ראשון שיוזם ה-Zygote. נוצר תהליך בן שירש את ה-ART, את ה-Framework, ובלית ברירה, גם את הסוקט. [cite: 71]

מתכנתי הקוד של אנדרואיד, הזדרזו לסגור את הסוקט מטעמי אבטחה, כפי שניתן לראות בקטע הקוד הבא: [cite: 72]

Zygote.closeAllFilesExcept(managedFileDescriptors); [cite: 73]

ה-Fork "נולד" עם מחרוזת בזיכרון, שתשמש אותו כדי להריץ את ה"ייעוד" שלו בעזרת פונקציות בזמן ריצה (ע"י מנגנון Reflection) [cite: 74]

מי שאחראי לכך שה-Fork הראשון ייווצר עם המחרוזת SystemServer.main(), זהו ה-Init (בכבודו ובעצמו). [cite: 79] הוא מעביר אותה כחלק מיצירת ה-Zygote עם דגל --start-system-server חד-פעמי (לחיסכון בזמן). [cite: 80] כך ה-Zygote מחזיק בזיכרון את המחרוזת ומיד משכפל את עצמו. [cite: 80, 81]

בהמשך, כאשר ה-System Server ירוץ ויגיד ל-Zygote שצריך Fork חדש, הוא זה שיקבל את המחרוזת מה-Launcher, ויעביר אותה ל-Zygote דרך ה-Local Socket. [cite: 82, 83]

מעתה והלאה, ה-Zygote עובר למצב האזנה בסוקט שפתח. מי שאחראי "להעיר" אותו וליזום קריאות של Zygote Forks, זה רק המנהל הראשי - ה-System Server. [cite: 84, 85]

חלוקת תפקידים [cite: 86]

תהליך ה-System Server [cite: 87]

הפונקציה SystemServer.main() היא זו ש"מייצגת" את מערכת ההפעלה, ומנהלת את תהליך האפליקציה. שניים מהשירותים המרכזיים שחשוב להכיר הם: [cite: 88]

  1. Activity Manager Service: ה-AMS, הוא ה"מבוגר האחראי" שמנהל את הרכיב המנוהל ActivityThread שבתוך האפליקציה. [cite: 89, 90]
  2. Package Manager Service: ה-PMS, אחראי לנהל ולשמור את המידע על כל האפליקציות המותקנות במכשיר. [cite: 91, 92]

ברגע שמתעורר ה-System Server במהלך ה-Boot, דבר ראשון שהוא עושה זה להריץ את ה-main שלו. [cite: 93] ה-PMS סורק את כל קבצי ה-Manifests שיש במכשיר (בפועל, סורק קובץ כללי אחד - כחלק מאופטימיזציה), ומעדכן טבלה פנימית שמגדירה הרשאות ושירותים לכל אפליקציה. [cite: 94]

מאוחר יותר, ה-Launcher (מסך הבית) רק "מתשאל" את ה-PMS, כדי לקבל את האייקון והשם (ה-Label) שלה, כדי להציג אותם במסך הבית (כפי שנראה בחלק ג', בקוד ה-XML הראשון). [cite: 95]

תהליך אפליקציה [cite: 96]

נוצר בכל פעם שהמשתמש לוחץ על ה-Launcher של האפליקציה. [cite: 97]

ה-System Server שולח פקודה ל-Zygote דרך ה-Socket: "תשכפל פרוסס חדש". התהליך החדש שנוצר, משתמש ב-Reflection כדי להריץ את ActivityThread.main(), שמפעיל את ה-Looper (נראה בהמשך) ומכין את ה-UI Thread לקבלת הודעות מה-System Server. [cite: 98, 99, 101]

ברגע שהאפליקציה "חיה ונושמת", ה-Zygote מסיים את תפקידו וחוזר "לישון". [cite: 105] מעתה והלאה, התקשורת השוטפת מתבצעת ישירות בין ה-ActivityThread לבין ה-System Server (ה-AMS), באמצעות מנגנון העברת הודעות מאובטח (IPC) הראשי של אנדרואיד - Binder IPC. [cite: 106]

תהליך ה-ActivityThread [cite: 111]

זהו "סוכן" שיושב בתהליך האפליקציה, ומקבל את הפקודות מ-AMS. למרות שמו, הוא אינו Thread, אלא מחלקת Java שמנהלת את האפליקציה מבפנים. [cite: 112] על מנת לתקשר עם החוץ, הוא מחזיק בתוכו מחלקה פנימית שמאזינה לבקשות מ-AMS. בקשות אלו מועברות לתור של ה-UI Thread. [cite: 113]

מנגנוני ה-Socket וה-Binder [cite: 114]

נמצאנו למדים, שיש ל-System Server כ-3 תפקידים עיקריים: [cite: 115]

  • לבקש בעצמו Zygote Forks עבור אפליקציות חדשות. [cite: 115]
  • לתקשר עם האפליקציה שנוצרה באמצעות ה-AMS. [cite: 116]
  • לסרוק את קובץ ה-AndroidManifest.xml במהלך ה-Boot או התקנת האפליקציה באמצעות ה-PMS. [cite: 118]

אך כאן עולה שאלה מעניינת: למה המערכת צריכה שני מנגנוני תקשורת (IPC) שונים? [cite: 119] בדומה למודלים של IPC בין דפדפן ליישומים בלינוקס, גם כאן התקשורת לא מתבצעת ישירות. [cite: 120]

  • בשלב ההקמה: שלב שבו עוברת התקשורת דרך ה-Local Socket, בין ה-System Server לבין ה-Zygote. [cite: 123] זהו צינור מהיר ופשוט שנועד למטרה אחת בלבד - להעיר את ה-Zygote ולהגיד לו לבצע שכפול. [cite: 124]
  • בשלב הריצה: שלב שבו עוברת התקשורת דרך ה-Binder, בין ה-AMS לבין ה-ActivityThread. [cite: 125]

מנוע החיים של התהליך: ה-Looper של ה-UI Thread [cite: 126]

(חשוב להבהיר: ארכיטקטורת אנדרואיד רוויה בלולאות Looper שונות, ולמעשה כמעט לכל Service). [cite: 127]

בהקשר של "לידת" אפליקציה, לאחר ה-Fork והרצת הפונקציה הייעודית מ-Framework, יש רק Thread אחד, ה-UI Thread. [cite: 128]

בפועל, כל מה שאנחנו קוראים לו 'האפליקציה', יושב על לולאה אחת (Looper) שחיה בתוך ה-UI Thread, שמנוהלת על ידי ה-ActivityThread. [cite: 129] היא משהה את ה-UI Thread מלהסתיים, וממתינה לפקודות ה-System Server כדי להריץ את קוד האפליקציה. [cite: 130]

כבר הזכרנו שה-System Server כבר מחזיק בזיכרון שלו הרשאות, וביניהן את ה-Entry Point של האפליקציה. [cite: 131] בזמן הריצה, ה-System Server שולח פקודות הרצה (כמו onCreate, onStart וכו'), כדי להניע את ה-Entry Point. [cite: 132] הפקודות האלו נשלחות מה-System Server דרך ה-Binder IPC, ומתורגמות להודעות בתוך תור ההודעות של האפליקציה. [cite: 133]

ה-UI Thread מצידו לא "נחנק" - הוא יושב בתוך לולאת ה-Looper האינסופית, שולף את הפקודות האלו מהתור אחת אחת בעזרת המתווך שלו, ומריץ את פונקציות האפליקציה בזו אחר זו. [cite: 134]

שתי נקודות הכניסה - Application vs. Activity [cite: 143]

באנדרואיד אין רק "נקודת כניסה" אחת. יש למעשה שתי רמות שונות של הרצת קוד שרצות זו אחר זו האחת ברמת ה"תהליך", והשניה ברמת ה"חלון." [cite: 144]

הזכרנו שה-System Server אחראי לשלוח פקודות ל-Looper. יש פקודה אחת שמתרחשת בתחילת הרצת האפליקציה, והיא מושפעת ממה שכתוב (או לא כתוב) ב-Manifest. [cite: 145] הפקודה הראשונה של ה-AMS נקראת bindApplication. היא הטריגר שגורם למערכת ההפעלה ליצור את אובייקט ה-Application (כפי שנראה בהמשך), והיא מתרחשת עוד לפני שהאפליקציה מריצה UI כלשהו. אתגר הניתוח הדינמי [cite: 221]

לאחר שהבנו שהנוזקה חוטפת את שלב ה-Bootstrap ורצה בתוך attachBaseContext(), עולה שאלה קריטית: איך בכלל אפשר לנתח אותה בזמן אמת? [cite: 222, 223] כאן נכנס לתמונה כלי ה-Dynamic Binary Instrumentation המוביל בתעשייה - Frida. [cite: 223, 224]

Frida הוא כלי המאפשר לחוקרים להזרים סקריפטים מותאמים אישית של JavaScript ישירות לתוך הזיכרון של תהליכים חיים, ובכך לבצע הזרקת סקריפט (Hooking) לפונקציות ולשנות את התנהגותן בזמן ריצה. [cite: 225] הבעיה היא הארכיטקטורה של הנוזקה, שדורשת מאיתנו לשנות את הדרך שבה אנחנו משתמשים ב-Frida. [cite: 226]

1. הגישה הרגילה והמכשול שלה: חיבור מאוחר(Attach) [cite: 231]

במצב הרגיל (Attach), אנחנו מנסים להתחבר לאפליקציה אחרי שהיא כבר הופיעה על המסך (בשלב ה-UI). [cite: 231] ההתחברות מתבצעת הלכה למעשה רק אחרי שהתחיל ה-Entry Point עם אובייקט ה-Activity. [cite: 232]

בשלב זה, Frida שולחת פקודה לקרנל: "עצור את ה-Threads של האפליקציה", מזריקה את סוכן הניטור שלה (ה-Frida Agent), ומשחררת את הריצה. [cite: 233] לרוע המזל, זה כבר מאוחר מדי. מתודת ה-attachBaseContext() של הנוזקה כבר רצה והסתיימה מזמן, יחד עם אובייקט ה-Application. [cite: 234]

2. הדרך לנצח את המירוץ: מצב Spawn [cite: 235]

מכיוון שאין לנו שום יכולת אנושית (או טכנית במצב Attach) להיות מהירים יותר מפקודות ה-Bootstrap של ה-System Server, הדרך לנצח היא להקדים את טעינת האפליקציה בעזרת מצב Spawn. [cite: 236]

במצב Spawn, האפליקציה מופעלת כרגיל (מה שגורם ל-System Server לבקש Fork מתוך ה-Zygote), אך Frida מתערבת מיד לאחר ה-Fork. [cite: 237] היא תופסת את התהליך החדש ומעבירה אותו למצב השהייה, עוד לפני שהפקודה הראשונה של ActivityThread מתחילה לרוץ (לפני ה-Process Entry Point). [cite: 237, 238]

כך, כשהנוזקה תגיע לבסוף אל ה-attachBaseContext(), אנחנו כבר נהיה שם בפנים כדי לתפוס אותה בזמן אמת. [cite: 239]

3. שבירת חוקי המשחק: Zygote Hook [cite: 240]

במקרים קיצוניים של נוזקות מתקדמות (או כאשר קיימות הגנות Anti-Analysis חזקות במיוחד), ניתן להשתמש בשיטה האגרסיבית ביותר: ה-Zygote Hook. [cite: 241] במקום להמתין ליצירת תהליך האפליקציה הספציפי, ה-Frida מזריקה את עצמה ישירות לתוך תהליך ה-Zygote המקורי עצמו. [cite: 242] המשמעות היא שכל אפליקציה עתידית שתרוץ במכשיר, תשתכפל ותיוולד כאשר ה-Frida Agent כבר נמצא ב-"DNA" שלה, עוד לפני שלב ה-Fork. [cite: 243]

ה-Hook במילים פשוטות: [cite: 244]
נניח שההוק שלי מתחיל לרוץ מיד אחרי ה-Fork, מה הוא רואה? [cite: 245]

  • שעוד מעט מתחילה סריקת ה-Manifest. [cite: 245]
  • הולך להיווצר אובייקט Application - ושם בדיוק נעצור אותו. [cite: 247]

נתפוס את מי שקרא ליצירת Application חדש, נכריח אותו ליצור אובייקט נקי, ונחזיר אותו ללא תלות בקוד הזדוני של האפליקציה. [cite: 247]

כדי להבין בדיוק מי יוצרת אובייקט חדש, חקרתי את הדרך שבה ה-ActivityThread בונה את האפליקציה, וגיליתי שהוא משתמש במחלקה פנימית בשם LoadedApk. [cite: 248] זיהיתי שהפונקציה makeApplicationInner (בתוך מחלקה זו) היא זו שקוראת פיזית ליצירת האובייקט (newApplication). [cite: 249, 250] היא מבצעת בדיקה פנימית על הפרמטר הראשון הבוליאני: [cite: 256]

if (forceDefaultAppClass || (appClass == null)) { [cite: 257, 258]
    appClass = "android.app.Application"; [cite: 259]
} [cite: 260]

[cite: 261] אך בגלל שהנוזקה כן קיימה והגדירה appClass זדוני, אנחנו נכריח את המשתנה forceDefaultAppClass להיות true בזמן ריצה, ומיד ייווצר אובייקט appClass חדש ונקי לחלוטין: [cite: 262]

השלכות למחקר הדינמי [cite: 334]

ברגע שהרצתי את ה-Hook, נוצר אובייקט Application בזכרון. אמנם כעת חסרה פונקציית הפענוח של קבצי ה-DEX שישבה בראש אובייקט ה-Application של הנוזקה (שדרסנו). [cite: 334] זה הוביל לתלונת ה-ClassLoader אודות קובץ ה-DEX האבוד. [cite: 335, 336]

בשלב המימוש, רציתי לזייף MySomeProvider תוך כדי ריצה ב-Frida, ולשלוח אותו ל-ClassLoader, כדי שלא יבכה שלא מצא אותו. [cite: 337] עם קצת ניסוי וטעיה, למדתי על הדינמיקה עם ה-ClassLoader. לא מספיק להזריק את ה-Class שלי ל-ART כי ה-ClassLoader מחפש Path של קובץ בדיסק הקשיח (ולא ב-RAM), ברגע טעינת ה-ActivityThread. [cite: 338, 339]

תיאור ה-Hook במילים: [cite: 340]

  1. ליצור SomeProvider.java ולקמפל ידנית ל-DEX. [cite: 342]
  2. לאתחל מופע חדש של DexClassLoader, שמירת ה-Path, הכנת MySomeProvider להזרקה ועדכון האב של ה-ClassLoader. [cite: 342]
  3. עצירת שרשרת האספקה והזרקת ה-MySomeProvider שיצרתי, לתוך תהליך קבלת ה-DEX ל-ART (לא הרחבתי, כדי לא לגלוש מגבולות המאמר). [cite: 343]
  4. כפיית אובייקט Default Application (כמו שכבר הכרנו). [cite: 344]

סיכום [cite: 395]

ראינו איך מכשיר האנדרואיד שלנו, כבר לא כ"כ בטוח כמו שחשבנו. המורכבות שלו יוצרת "פתחים", שנוזקות מתקדמות יודעות לנצל. אותי זה הביא למסקנה - להשתמש בו לצרכי מחקר ופחות לצרכים אישיים ושימוש יומיומי (ולהעדיף פלאפון מקשים). [cite: 396]

מערכת מורכבת כמו אנדרואיד, מלאה ב-APIs שמקשרים לעוד APIs, והמון דברים קורים "מתחת לפני השטח". [cite: 397]

המחקר הוביל למסקנה, שבאנדרואיד אין "נקודת כניסה" אחת לאפליקציה, אלא שרשרת Bootstrap מרובת שלבים. [cite: 398] כל אחד מהשלבים מהווה שטח תקיפה בפני עצמו עבור נוזקות. [cite: 399]

מה שריגש אותי במחקר, הוא לראות איך שבריר שנייה בודד בזמן הריצה, יכול להרגיש כמו "אינסוף", של שעות ארוכות מול המסך. [cite: 400] כמו שכתוב בתורה, "יום לשנה" - יש דברים שקרו ביום אחד, ולוקח שנה לספר אותם. [cite: 401, 402]
