Breaking the Compilation Boundary: Semantic Fuzzing of the SBCL Runtime

יולי 2025 | קטגוריה: Vulnerability Research, Compiler Internals
סטטוס: Responsible Disclosure מול CERT-IL (מס' 11265) ו-MITRE.
תקציר
במהלך מחקר Fuzzing על קומפיילר SBCL (שפת LISP), התמקדתי בשלב פריסת המאקרו (Macro Expansion). המטרה הייתה לבחון האם ניתן לשבור את מנגנון ה-Lexical Scoping בזמן קומפילציה. תוך כדי המחקר, ששילב אוטומציה בעזרת AI Agent, זיהיתי כשל לוגי המאפשר לעקוף את הבידוד הלקסיקלי ולהזריק משתנים דינמיים אל מחוץ ל-Scope ה"מוגן".

1. הגדרת הבעיה

השאלה המרכזית שעמדה בבסיס המחקר היא כיצד הקומפיילר מתמודד עם גבולות הבידוד של משתנים לקסיקליים בזמן פריסת מאקרו. מכיוון שמאקרואים משנים קוד לפני יצירת ה-Stack Frame, ניסיתי לבדוק האם מניפולציה בשלב הזה יכולה לגרום למשתנה לדלוף או לשנות את המצב הלוגי שלו בסביבה המקומית.

2. מתודולוגיית הבדיקה: Grammar-Aware Fuzzing

מכיוון שקומפיילרים קורסים מיד על קלט אקראי בשל שגיאות Syntax, אי אפשר להשתמש ב-Fuzzing מסורתי כדי למצוא באגים לוגיים. לשם כך, נבנה Harness אוטומטי שמתמקד בסמנטיקה:

  • Grammar-Aware Mutation: ה-Fuzzer הוגדר לייצר עצי תחביר (AST) חוקיים בלבד, תוך שימוש בדקדוק תקני, כדי לצלוח את שלבי ה-Lexing וה-Parsing.
  • Semantic Oracle: במקום לחפש קריסות (Segfaults), המערכת עקבה אחרי אנומליות לוגיות באמצעות stdout. הפריצה סומנה ברגע שמשתנה בסביבה לקסיקלית סגורה שינה את ערכו בעקבות הרצת המאקרו, מבלי שהקומפיילר יזהה שגיאה או יעצור את ההידור.

3. ממצאים ארכיטקטוניים והשוואה למודל של Java

כדי להבין את מהות הכשל ב-SBCL, שווה להשוות אותו למודל הטיפוסיות ב-Java Generics, ספציפית לעקרון PECS והשימוש ב-? super T.

ב-Java, הקומפיילר נוקט בגישה פסימית קשיחה: כל ספק לגבי בטיחות הטיפוסים גורר שגיאת קומפילציה. לדוגמה, ברשימה מסוג List<? super Apple>, הקומפיילר אוסר על הכנסת אובייקט מסוג האב Fruit (דרך add), מכיוון שיש סיכוי שהרשימה מכילה רק תפוחים. ההחמרה הזו נועדה למנוע זיהום של הזיכרון וקריסות בזמן ריצה.

ב-SBCL המצב שונה. הקומפיילר לא מוודא את תקינות הפעולה מול הסביבה הלקסיקלית (ה-LEXENV) בזמן הרחבת המאקרו. הדבר איפשר לי לבודד שלושה וקטורי תקיפה שונים:

וקטור 1: הזרקת קישור דינמי (Dynamic Binding Injection)

וקטור זה מנצל פער בסדר הפעולות (Pass Order) של הקומפיילר. להלן תרשים הממחיש את שלבי ההידור ואת נקודת הזרקת הקוד:

[Step 1: Front-End] Read & Parse (AST Generation)
                      |
                      v
[Step 2: Lexical Analysis] Register "secret-token" as LOCAL in LEXENV
                      |
                      v
[Step 3: Macro Expansion] -> Runs inject-dynamic-binding macro
                           * Evaluates: (proclaim '(special secret-token))
                           * Global Symbol Table is OVERRIDDEN immediately!
                      |
                      v
[Step 4: Code Generation] Compiler checks Symbol Table for "secret-token"
                           * Sees dynamic SPECIAL_FLAG set in Step 3
                           * CRITICAL BYPASS: Generates Dynamic Runtime Lookup!
                      |
                      v
[Step 5: Back-End] Final Assembly/Bytecode generation (Isolation broken)

מה בעצם קורה כאן?
בשלב 2, המהדר רושם את המשתנה secret-token כמשתנה מבודד (Lexical) בתוך בלוק ה-let. אולם בשלב 3, במהלך פריסת המאקרו, פקודת ה-proclaim משנה את המטא-דאטה של הסמל בטבלת הסמלים הגלובלית, ומגדירה אותו כ-SPECIAL (דינמי).

כאשר המהדר מגיע לשלב 4 (Code Generation), הוא בוחן שוב את טבלת הסמלים, מזהה את הדגל הגלובלי החדש שנשתל, ומייצר קוד המבצע קריאה דינמית (Dynamic Runtime Lookup) במקום הפניה לקסיקלית מאובטחת. גבולות ה-Scope המקומי נדרסים ברמת הבינארי, וקוד חיצוני יכול כעת לגשת למשתנה.

;; PoC Code: dynamic_binding_attack.lisp
(defmacro inject-dynamic-binding (var-name)
  (proclaim `(special ,var-name)) ; Immediate mutation of global symbol metadata inside compilation phase
  `(inline-log "Flag compilation modified"))

(defun target-function ()
  (let ((secret-token "ENCRYPTED_DATA")) ; Supposed to be immutable and isolated lexically
    (inject-dynamic-binding secret-token) ; Triggering macro expansion and flag override
    secret-token)) ; Compiler now generates dynamic runtime lookup instead of safe lexical offset

;; Under vulnerable compilers, external call stack manipulation can now hijack 'secret-token'

וקטור 2: זיהום טבלת הסמלים (Global Symbol Table Pollution)

שימוש בפונקציית intern במהלך פריסת המאקרו ליצירת מצביעים כפולים. המהדר נכשל בבידוד ההפניות, מה שמוביל ליצירת Reference חופף בין האובייקט הגלובלי למצביע הלקסיקלי. פגיעות זו מובילה להדלפת מידע (Information Disclosure) מתוך סקופים חסומים.

;; PoC Code: symbol_interning_attack.lisp
(defmacro pollute-symbol-table (sym-name)
  (let ((global-sym (intern (string sym-name))))
    (setf (get global-sym 'compromised-leak) t)
    `(quote ,global-sym)))

(defun secure-enclave ()
  (let ((internal-key "PRIVATE_KEY"))
    (pollute-symbol-table internal-key) ; Context confusion during compilation
    ;; Execution trace leaks internal-key bounds to global symbol property listings
    ))

וקטור 3: בלבול מרחבי שמות (Cross-Package Namespace Confusion)

על ידי יצירת חבילה (Package) דינמית ב-Compile-time ושימוש בפקודת shadow, ניתן לשבור את טבלת המעקב של הקומפיילר אחר משתנים מקומיים, ולנתב את הגישה אליהם ישירות למרחב הכללי (cl-user).

;; PoC Code: package_hijack_attack.lisp
(defmacro hijack-namespace (target-sym)
  (let ((shadow-pkg (make-package "EPHEMERAL-ATTACK-PKG" :use '("CL"))))
    (shadow (string target-sym) shadow-pkg) ; Hijacking compilation target pointer
    `(inline-log "Namespace tracing broken")))

(defun sensitive-scope ()
  (let ((auth-token "SESSION_VALID"))
    (hijack-namespace auth-token) ; Compilation context dereferences token from local frame
    auth-token))

4. מודל איום: פגיעה בשרשראות אספקה (Supply Chain)

וקטורים מסוג זה מהווים סיכון משמעותי לתשתיות CI/CD. תוקף יכול לפרסם חבילת קוד המכילה "Poisoned Macro". בעת קימפול הפרויקט בשרת ה-Build, המאקרו מופעל, פורץ את גבולות הסקופ שלו, ויכול לגשת למשתנים רגישים (כגון מפתחות API או סודות הצפנה) של רכיבים אחרים בסביבת העבודה, כל זאת ללא זיהוי או קריסה בזמן הריצה השוטף.

🛠️ שלב מחקר נוכחי: ניתוח סטטי (IDA Pro)

לאחר אימות הפגיעות ברמת הקוד (Black-box), אני מנתח כעת את קובץ הבינארי של הקומפיילר (sbcl.exe) באמצעות IDA Pro.

המטרה היא למפות את ה-Cross-References הרלוונטיים באמצעות סקריפטים ב-IDAPython, לאתר את ההוראות (Assembly Instructions) שמבצעות את השינוי לדגל SPECIAL_FLAG מבלי לבצע וולידציה מול ה-Scope, ולבודד את ה-Root Cause ברמת האסמבלי.
