Breaking the Compilation Boundary: Semantic Fuzzing of the SBCL Runtime

יולי 2025 | קטגוריה: Vulnerability Research, Compiler Internals
סטטוס: Responsible Disclosure מול CERT-IL (מס' 11265) ו-MITRE.
תקציר:
במהלך מחקר fuzzing שביצעתי על הקומפיילר של שפת LISP, במטרה למצוא שבירה בתהליך פריסת ה-MACRO בזמן קומפילציה, זיהיתי כשל לוגי במנגנון ה-Lexical Scoping. החולשה מאפשרת לעקוף את הבידוד הלקסיקלי, ולהזריק משתנים דינמיים מחוץ ל-Scope ה"מוגן". המחקר בוצע בשילוב תהליכי אוטומציה בעזרת AI Agent Copilot.

1. שאלת המחקר

כיצד הקומפיילר, בזמן פריסת ה-Macro בקוד, שומר על גבולות הבידוד של משתנים לקסיקליים בסקופים שונים? והאם ניתן לגרום למשתנים מחוץ למתחם ה-Macro לגלוש אליו ולשנות את מצבו הלוגי?

2. הניסוי: בניית ה-Grammar-Aware Fuzzer

קומפיילרים הם מערכות קשיחות; קלט אקראי ייעצר ב-Lexer או ב-Parser על שגיאות תחביר (Syntax). כדי להגיע לבאגים לוגיים עמוקים, נבנה מערך בדיקות אוטומטי (Harness):

  • Grammar-Aware Mutation: ה-Fuzzer ייצר עצי תחביר (AST) חוקיים בלבד מבחינת דקדוק וסוגריים, כדי לאפשר מעבר חלק של שלבי ההידור הראשוניים.
  • Semantic Oracle: מערך הבדיקה לא התמקד בקריסות זיכרון (Segfaults), אלא באנומליות לוגיות. בתהליך של פינג-פונג מהיר מול המהדר, המערכת עקבה אחרי הפלט (stdout) וסימנה פריצה מושגת ברגע שמשתנה שאמור להיות חסום ומבודד שינה את ערכו עקב פעולת המאקרו – ללא התרעה או שגיאה מצד הקומפיילר.

3. שלושת הממצאים (Vulnerability Primitives) וההשוואה ל-Java Generics

מנגנון הבדיקה הציף פער ארכיטקטוני חריף, במיוחד כאשר משווים אותו למנגנון ה-Wildcards ב-Java Generics, ותחת עקרון PECS עם הביטוי ? super T.

ב-Java, הקומפיילר פועל בצורה פסימית ומחמירה כדי למנוע כשלים לוגיים בזמן ריצה. אם ישנו ספק לגבי בטיחות הפעולה, הוא יחסום אותה ויזרוק שגיאת קומפילציה. ברשימה המוגדרת כ-List<? super Apple> basket, המהדר מבין שמדובר ברשימה של תפוחים או אבותיו (כמו Fruit או Object). מאחר שהוא אינו יודע מהו הטיפוס המדויק, ניסיון להכניס אובייקט מסוג האב הכללי Fruit באמצעות basket.add(new Fruit()) ייחסם מיד בשגיאת קומפילציה, מחשש שמא מדובר בפועל ברשימת תפוחים ספציפית וההכנסה תגרום לזיהום הטיפוסים. בקריאה (get), המהדר יגן על גבולות המשתנה ויאפשר שליפה אך ורק לתוך הטיפוס הכללי והבטוח ביותר – Object.

במהדר SBCL, המציאות הפוכה. במקום לנהוג בפסימיות, לחשוש מפני זיהום לוגי ולחסום את הפעולה, המהדר מאפשר מניפולציות compile-time עקומות ללא אימות תקינות מול ה-LEXENV (הסביבה הלקסיקלית האקטיבית). להלן ניתוח וקטורי התקיפה הלוגיים בצירוף דוגמאות הקוד (PoC) שייצר ה-Fuzzer:

וקטור 1: הזרקת קישור דינמי (Dynamic Binding Injection)

וקטור זה מנצל כשל לוגי בסדר פעולות עיבוד הנתונים של המהדר (Compilation Pass Order). כדי להבין כיצד פקודת המאקרו מצליחה לשבור את ה-Scope המקומי, נביט בצינור ההידור (Pipeline) של המהדר והיכן נקודת הזיהום מתרחשת:

[Step 1: Front-End] Read & Parse (AST Generation)
                      |
                      v
[Step 2: Lexical Analysis] Register "secret-token" as LOCAL in LEXENV
                      |
                      v
[Step 3: Macro Expansion] -> Runs inject-dynamic-binding macro
                           * Evaluates: (proclaim '(special secret-token))
                           * Global Symbol Table is OVERRIDE immediately!
                      |
                      v
[Step 4: Code Generation] Compiler checks Symbol Table for "secret-token"
                           * Sees dynamic SPECIAL_FLAG set in Step 3
                           * CRITICAL BYPASS: Generates Dynamic Runtime Lookup!
                      |
                      v
[Step 5: Back-End] Final Assembly/Bytecode generation (Isolation broken)

הסבר המנגנון בקצר:
כפי שניתן לראות בתרשים, בשלב 2 המהדר פועל בצורה תקינה ורושם את המשתנה secret-token כמשתנה מקומי ומבודד (Lexical) בתוך בלוק ה-let. בשלב זה, הערבויות של השפה עדיין מתקיימות.

הכשל מתרחש במעבר לשלב 3 (Macro Expansion). המהדר עוצר את זרימת ההידור הסטנדרטית ומריץ את פונקציית המאקרו. בתוך המאקרו, פקודת ה-proclaim מופעלת ומשנה את ה-Metadata של הסמל באופן מיידי וגלובלי בטבלת הסמלים (Global Symbol Table), והופכת אותו ל-SPECIAL (דינמי).

כאשר המהדר מסיים את הרחבת המאקרו ועובר לשלב 4 (Code Generation) כדי לייצר קוד עבור המשך הפונקציה, הוא ניגש לבדוק כיצד לתרגם את הגישה למשתנה secret-token. המהדר מזהה את הדגל הגלובלי החדש שנשתל בשלב 3, מתבלבל סמנטית, ומשנה את ארכיטקטורת הפלט: במקום לייצר הפניה לקסיקלית בטוחה על ה-Stack Frame, הוא מייצר קוד של קישור דינמי (Dynamic Runtime Lookup) מבוסס מחסנית קריאות. גבולות הבידוד של ה-Scope המקומי נמחקו לחלוטין ברמת הבינארי הסופי, וקוד חיצוני יכול כעת לקרוא ולדרוס את המשתנה.

;; PoC Code: dynamic_binding_attack.lisp
(defmacro inject-dynamic-binding (var-name)
  (proclaim `(special ,var-name)) ; Immediate mutation of global symbol metadata inside compilation phase
  `(inline-log "Flag compilation modified"))

(defun target-function ()
  (let ((secret-token "ENCRYPTED_DATA")) ; Supposed to be immutable and isolated lexically
    (inject-dynamic-binding secret-token) ; Triggering macro expansion and flag override
    secret-token)) ; Compiler now generates dynamic runtime lookup instead of safe lexical offset

;; Under vulnerable compilers, external call stack manipulation can now hijack 'secret-token'

וקטור 2: זיהום טבלת הסמלים (Global Symbol Table Pollution)

יצירת מצביעים כפולים או כפייה של מיפוי מחדש באמצעות פונקציית intern בזמן פריסת המאקרו. המהדר נכשל בשמירה על ההפרדה הלוגית (Isolation), ויוצר קישור פנימי דרוס (Collided Reference) בין ה-Global Object Storage לבין ה-Lexical Pointer, מה שמאפשר להדליף ערכים חסומים החוצה (Information Disclosure).

;; PoC Code: symbol_interning_attack.lisp
(defmacro pollute-symbol-table (sym-name)
  (let ((global-sym (intern (string sym-name))))
    (setf (get global-sym 'compromised-leak) t)
    `(quote ,global-sym)))

(defun secure-enclave ()
  (let ((internal-key "PRIVATE_KEY"))
    (pollute-symbol-table internal-key) ; Context confusion during compilation
    ;; Execution trace leaks internal-key bounds to global symbol property listings
    ))

וקטור 3: בלבול מרחבי שמות בין חבילות (Cross-Package Namespace Confusion)

שימוש בטכניקת shadow או מניפולציות של Package Systems מתוך חבילה דינמית שנוצרה ב-Compile-time. הפעולה גורמת לקריסה של ה-Trace שהמהדר מנהל עבור המשתנה המקומי, ומנתבת את הגישה אליו למרחב הציבורי הכללי (cl-user).

;; PoC Code: package_hijack_attack.lisp
(defmacro hijack-namespace (target-sym)
  (let ((shadow-pkg (make-package "EPHEMERAL-ATTACK-PKG" :use '("CL"))))
    (shadow (string target-sym) shadow-pkg) ; Hijacking compilation target pointer
    `(inline-log "Namespace tracing broken")))

(defun sensitive-scope ()
  (let ((auth-token "SESSION_VALID"))
    (hijack-namespace auth-token) ; Compilation context dereferences token from local frame
    auth-token))

4. מודל איום (Supply Chain) – למה זה מסוכן?

החולשה אינה דורשת הרשאות מנהל, ומציגה וקטור תקיפה מובהק עבור שרתי בנייה אוטומטיים (CI/CD Pipelines). תוקף יכול לשתול מאקרו זדוני ("Poisoned Macro") בתוך חבילת קוד צד-שלישי. בזמן קומפילציית הפרויקט, המאקרו מופעל בשלב הבנייה, חורג מגבולות הסקופ המבודד שלו, ומסוגל לשלוף או לשכתב משתנים וערכים רגישים של רכיבים אחרים בפרויקט (כמו מפתחות הצפנה או API Tokens) – מבלי לעורר התרעות בזמן הריצה הסטנדרטי.

🛠️ השלב הבא: ניתוח סטטי ב-IDA Pro (עבודה בעיצומה)

לאחר הוכחת הכשלים הלוגיים באמצעות Black-box Fuzzing, שלב המחקר הנוכחי (Work in Progress) מתמקד בניתוח סטטי של קובץ הבינארי (sbcl.exe) ב-IDA Pro.

העבודה מבוצעת באמצעות סקריפטים ב-IDAPython במטרה למפות את ה-Cross-References לפונקציות הניהול של ה-Namespaces, לאתר את פקודות האסמבלי המדויקות שמבצעות את השינוי הבינארי לדגל ה-SPECIAL_FLAG ללא וולידציה, ולהצביע בדיוק על שורות ה-Assembly שמהוות את ה-Root Cause של הכשל המבני.

ממצאי ה-Assembly המלאים יעודכנו כאן עם סיום השלב.
