MyLeads AI — Security Architecture Case Study (AppSec Perspective)

Secure-by-Design SaaS with CI/CD Security, Zero Trust Infrastructure & AI Integration

מאת שי מרדכי | 9 במרץ 2026

0. Executive Summary

MyLeads AI היא מערכת SaaS מבוססת AI לניהול לידים, שנבנתה מתוך הנחה אחת מרכזית:

כל מערכת חשופה לאיום כברירת מחדל — ולכן אבטחה חייבת להיות חלק מהארכיטקטורה, לא תוספת.

המערכת תוכננה סביב שלושה עקרונות:

  • Secure SDLC מלא עם CI/CD אוטומטי
  • Zero Trust Cloud Architecture
  • AI-powered application workflows עם מודעות ל-abuse scenarios

איור 1: ארכיטקטורת הייצור Zero Trust

1. Threat Model Overview

בשלב התכנון זוהו שלוש קטגוריות סיכון מרכזיות:

1.1 External Attack Surface

  • Internet-facing SaaS application
  • Automated bot scanning
  • API abuse attempts

1.2 Application Layer Risks

  • Authentication bypass
  • IDOR / Broken Access Control
  • Session hijacking
  • Webhook spoofing
  • Input-based injection vectors

1.3 AI & Automation Risks

  • Prompt injection
  • Data leakage via AI outputs
  • Misuse of AI-generated business decisions

2. Secure SDLC & CI/CD Security Model

המערכת נבנתה סביב pipeline מלא של:

  • GitHub Actions (build/test/deploy orchestration)
  • Pytest (backend validation)
  • Vitest + Playwright (frontend + E2E validation)
  • Docker-based containerization
  • Deployment to AWS ECR + EC2

Security Design Principle: ה-pipeline אינו רק כלי DevOps — אלא security enforcement layer. כל שינוי עובר validation לפני build artifact generation, קיימת הפרדת backend/frontend pipelines לצמצום blast radius, ונמנעת פריסה (deployment) של state לא מאומת.

3. Zero Trust Cloud Architecture

3.1 Network Exposure Model

המערכת נבנתה כך שאין פתיחת inbound ports ל-EC2. כל התעבורה נכנסת דרך Cloudflare Tunnel בלבד (outbound-only model), וגישה ניהולית מוגבלת מתבצעת דרך Cloudflare Access + MFA.

→ המשמעות: אין IP ציבורי שניתן לתקוף ישירות.

3.2 Runtime Isolation

  • Containers מופעלים ב-rootless Podman.
  • הפרדת services (backend / worker / frontend).
  • הגבלת resource allocation לכל שירות.

4. Secrets & Identity Security

4.1 Secret Management Model

במקום שימוש ב-.env סטטי, secrets נטענים בזמן runtime מתוך AWS SSM Parameter Store ללא persistence על דיסק.

→ הפחתת סיכון של: LFI, SSRF-to-file exfiltration, ו-repo leakage.

4.2 AWS Metadata Protection

שימוש ב-IMDSv2 בלבד (token-based metadata access) המפחית דרמטית סיכוני SSRF metadata abuse.

5. Authentication & Session Security Design

5.1 Token Model

שימוש ב-httpOnly JWT cookies בלבד למניעת client-side token exposure (XSS), והפרדה מוחלטת בין session state ל-business logic state.

5.2 Authorization Model

כל החלטת הרשאה מבוססת JWT claims בלבד. אין הסתמכות על user-controlled identifiers, וישנה ריכוזיות (centralization) של get_current_user logic.

→ Mitigates: IDOR, privilege escalation, ו-broken object-level authorization.

6. AI Security Considerations

המערכת כוללת רכיבי AI ל-lead scoring, conversational simulation, ו-workflow automation.

  • Threats Considered: Prompt injection, data poisoning, ושימוש לרעה ב-AI outputs לטובת business logic.
  • Mitigation Approach: הפרדה מוחלטת בין AI output לבין החלטות קריטיות במערכת. ה-AI מוגדר ככלי "תומך החלטה" בלבד (untrusted subsystem), עם הגבלת scope מחמירה של פעולותיו.

7. Application Security Controls

7.1 Input & Access Control

Validation מתבצעת Server-side בלבד, תוך אכיפת strict API boundary enforcement ומניעת אמון (Trust) ב-client input.

7.2 Web/API Security

Mitigation מובנה ברמת הארכיטקטורה כנגד: XSS, IDOR, webhook spoofing, ו-path traversal (Design awareness).

8. Observability & Audit Logging

שימוש ב-structured JSON logs. מודל dual-write logging (שמירה ל-DB + פורמט המוכן ל-AWS CloudWatch). חשוב מכך: failure-safe audit pipeline (שגיאת רישום לוג אינה מפילה את השירות ואינה מאפשרת DoS דרך מנגנון הלוגים).

→ מונע: Log tampering ו-audit-based DoS.

9. Key Security Engineering Decisions

  • Decision 1: Edge-first security (Cloudflare Tunnel) → Reduce attack surface before application layer.
  • Decision 2: Runtime secrets (SSM) → Eliminate static credential exposure.
  • Decision 3: CI/CD as enforcement layer → Prevent insecure builds from reaching production.
  • Decision 4: AI treated as untrusted subsystem → Avoid business logic dependency on AI output.

10. Conclusion

MyLeads AI מדגים גישה של: Security as Architecture, not as Post-Deployment Hardening.

המערכת נבנתה סביב עקרונות תשתיתיים של Zero Trust, Least Privilege, Secure SDLC, Runtime isolation, ו-AI-aware security design.
