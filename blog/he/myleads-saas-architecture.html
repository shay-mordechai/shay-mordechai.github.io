פרויקט MyLeads AI: ארכיטקטורת SaaS מאובטחת by Design

מאת שי מרדכי | 9 במרץ 2026

כשמפתחים פלטפורמת SaaS מודרנית שמטפלת בפניות לקוחות רגישות ובאינטראקציות AI, אבטחה לא יכולה להיות מחשבה שניה. מאמר זה מתאר את ההחלטות הארכיטקטוניות ואת צינורות ה-DevSecOps מאחורי MyLeads AI — פלטפורמה שתכננתי לפי פילוסופיית "Secure-by-Design" קפדנית.

המטרה הייתה לחסל משטחי תקיפה חיצוניים תוך מתן חוויית משתמש בעלת ביצועים גבוהים, תוך שימוש אינטנסיבי ב-AI לצורך מהירות פיתוח — כדי שאוכל להתמקד לחלוטין בהקשחת תשתיות ומידול איומים.

איור 1: ארכיטקטורת הייצור Zero Trust

0. איך זה נבנה: פיתוח AI-First

לפני שנצלול לארכיטקטורה, חשוב להיות שקופים לגבי גישת הפיתוח — כי היא חלק בלתי נפרד ממה שהפך את הפרויקט הזה לאפשרי.

MyLeads AI נבנתה כולה דרך פיתוח בסיוע AI, בשלושה שלבים מובחנים שמשקפים את ההתפתחות של המוצר עצמו:

שלב 1 — GitHub Copilot (VS Code): הגרסה הראשונית של הפלטפורמה נבנתה עם GitHub Copilot בתוך VS Code. בשלב זה המוצר היה פרוטוטיפ: לכידת לידים בסיסית, אותנטיקציה פשוטה, וממשק צ'אט ראשוני. Copilot האיץ את העבודה הכבדה בשלבים המוקדמים — ניתוב, מודלים, נקודות API — בעוד שאני התמקדתי בהחלטות הארכיטקטורה והאבטחה.

שלב 2 — Cursor: אחרי כשנה, קיבלתי החלטה מכוונת לשדרג את כל תהליך הפיתוח ל-Cursor. זה פתח מצב עבודה שונה מהיסוד: במקום השלמה אוטומטית, ניהלתי שיחות ארכיטקטוניות מלאות עם הקוד. Cursor אפשר לי לבצע refactor ממונוליט למונו-ריפו נקי של Next.js + Python, לממש את ה-WASM DLP Firewall, ולהגר את שכבת מסד הנתונים ל-Alembic — הכל בקצב שלא היה אפשרי בעבודה שורה אחר שורה.

שלב 3 — AI Studio (פיצ'רים מונחי לקוח): השלב האחרון היה מונע על ידי המוצר. עבדתי ישירות מול לקוח אמיתי — בעל עסק קואצ'ינג — ותכננתי דרישות בזמן אמת. כל פיצ'ר חדש (שידורי קול, כישורי AI לאיכות לידים, פורטל שותפי סוכנות) הוגדר בשיחה עם הלקוח, תורגם לפרומפט מדויק, ומומש דרך AI. Google AI Studio הפך לכלי הבחירה לפרוטוטייפינג מהיר של ה-flows האג'נטיים מבוססי Gemini.

תובנת פיתוח: המיומנות האמיתית בפיתוח בסיוע AI היא לא לכתוב פחות — אלא לדעת בדיוק מה לבקש, לזהות מתי הפלט שגוי, ולקבל החלטות ארכיטקטוניות שאף AI לא יקבל במקומך. כל החלטת אבטחה בקוד הזה התקבלה על ידי בן אדם. ה-AI כתב את הקוד.

1. תשתית "השרת האפל"

הרעיון המרכזי של סביבת הייצור הוא אי-נראות. שרת ה-AWS EC2 חושף אפס פורטים נכנסים (0.0.0.0/0 חסום) לאינטרנט הציבורי. כל תעבורת הכניסה מנותבת ומבוקרת אך ורק דרך רשת ה-Edge של Cloudflare.

  • Identity-Aware Proxy (IAP): גישת SSH מתבצעת אך ורק דרך פרוקסי של cloudflared ומחייבת אימות עם מפתח חומרה.
  • בידוד קונטיינרים: סטק האפליקציה רץ על Red Hat Enterprise Linux (RHEL 10.1) עם Rootless Podman מנוהל על ידי Systemd במרחב המשתמש, מה שמצמצם משמעותית את רדיוס הפיצוץ במקרה של פריצת קונטיינר.

2. חומת אש WASM למניעת דליפת מידע (DLP)

כדי למנוע API Over-fetching וחשיפת מידע ברמת האפליקציה (CWE-209), מימשתי פילטר WebAssembly (WASM) מותאם אישית שנכתב ב-Rust, שרץ ישירות בתוך Envoy Proxy.

חומת האש הזו משתמשת בארכיטקטורת Fail-Closed. היא בוחנת payloads של JSON יוצאים ומעמעמת בזמן אמת אישורים רגישים (כמו טוקנים פנימיים), כדי להבטיח שדליפת מידע נעצרת בקצה הרשת לפני שמגיעה ל-Frontend של Next.js.

3. ניהול Out-of-Band וה"כרית האוויר"

פאנלי ניהול הם יעדים ידועים לפרצות אינטרנט. כדי לצמצם את הסיכון:

  • ניהול דרך CLI בלבד: פעולות אדמין מושבתות בממשק האינטרנט ומועברות לכלי CLI מאוחד ומאובטח שנגיש רק דרך SSH.
  • Global Exception Handler: המערכת מיירטת שגיאות 500 Internal Server Error, מסירה stack traces למניעת דליפה, מציגה הודעה גנרית ומקומית, ושולחת דוח תקלות HTML מפורט לצוות ההנדסה מחוץ לערוץ הרגיל.

4. שלמות נתונים ועבודה מקבילית

טיפול באירועי webhook מקבילים בהיקף גבוה מפלטפורמות כמו Meta מחייב בטיחות טרנזקציונית. הגדרתי SQLite עם Write-Ahead Logging (WAL) למניעת שגיאות נעילה. בנוסף, Dead Letter Queue (DLQ) מבטיח שפayloads כושלים (למשל בגלל timeout של API ל-AI) מנותבים בבטחה לשחזור ידני — ומבטיח 0% אובדן מידע במהלך Retry Storms.

5. אינטגרציית AI: מקומי מול ענן

הפלטפורמה כוללת מנוע AI היברידי. רכיבי שמע רגישים (כמו הקלטות קואצ'ינג) מעובדים לחלוטין אופליין באמצעות מודל Faster-Whisper מקומי ו-FFmpeg מקומפל נטיבי. משימות אג'נטיות פונות לציבור (זיכרון שיחה, כישורי לידים) מועברות בצורה מאובטחת ל-Google Gemini 2.0 Flash עם מטמון Redis סמנטי למזעור זמן תגובה.

תובנה ארכיטקטונית: פיתוח מודרני כבר לא עוסק רק בכתיבת לוגיקה עסקית. על ידי שימוש בכלי AI — Copilot, Cursor, ו-AI Studio — לטיפול בייצור הקוד בכל שלב, הצלחתי להקדיש את משאבי ההנדסה שלי למה שבאמת חשוב: תכנון תשתית חזקה, עמידה בפני תקלות, ומאובטחת ברמה ארגונית. ה-AI כתב את המימוש. אני תכננתי את המערכת.
