Breaking the Compilation Boundary: Semantic Fuzzing of the SBCL Runtime

By Shay Mordechai | July 31, 2025

Status: Responsible Disclosure with CERT-IL (#11265) and MITRE.

Executive Summary: During a focused research on the Common Lisp compiler architecture, specifically the SBCL (Steel Bank Common Lisp) runtime environment, I identified 3 architectural logic flaws in the Lexical Scoping mechanism. These vulnerabilities allow an attacker to bypass the language's static isolation and inject Dynamic Bindings directly out of the protected Scope during Macro Expansion. The research was conducted using a custom Grammar-Aware Semantic Fuzzer.

1. The Architectural Hypothesis

It all started from an academic project in the Nand2Tetris course, where I built a custom compiler. While investigating Meta-programming capabilities across different languages, a theoretical research question arose:

Since Macros modify and inject code at compile-time — before a Runtime Stack Frame is ever created — what internal compiler mechanism ensures that local (lexical) variables remain completely isolated and uncontaminated by external code during expansion?

To test this theory in reality, I chose to attack SBCL (v2.3.2) — a commercial, veteran, and complex compiler.

2. Semantic Fuzzing & The AI Harness

Compilers crash immediately on Syntax errors against standard Fuzzing attacks. To find deep logic bugs, I built a focused Test Harness together with an AI Agent:

  • Grammar-Aware Mutation: The Fuzzer generated strictly valid Abstract Syntax Trees (AST), creating nested environments and modifying the Symbol Table at compile-time.
  • Semantic Oracle: The system didn't look for memory crashes (Segfaults), but rather logic flaws: a state where an isolated variable changed its value due to the macro operation without the compiler throwing an error.

3. The Vulnerability Primitives

The process yielded 3 different vectors where the compiler failed to validate the LEXENV (Active Lexical Environment):

Primitive 1: Dynamic Binding Injection

Using proclaim from within the macro expansion phase causes the compiler to globally update the symbol's flags. If the symbol is concurrently inside a protected let block, the resolution path shifts to the dynamic environment, overwriting the lexical memory.

Primitive 2: Global Symbol Table Pollution

Creating double pointers using intern during macro expansion. The compiler failed to maintain isolation, enabling the leakage of protected values (Information Disclosure).

Primitive 3: Cross-Package Namespace Confusion

Using the shadow technique from within a dynamic package created at compile-time caused the loss of the lexical trace, routing protected variables into the public namespace.

4. Threat Model (Supply Chain)

The primary risk here impacts CI/CD pipelines. An attacker could publish an innocent-looking third-party code package containing a "Poisoned Macro". When the build server compiles the project, the macro breaches its isolated scope and can extract or overwrite variables belonging to other developers in the same project (such as keys or tokens) — without triggering any standard runtime alerts.
