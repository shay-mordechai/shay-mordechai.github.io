Breaking the Compilation Boundary: Semantic Fuzzing of the SBCL Runtime

By Shay Mordechai | July 2025 | Category: Vulnerability Research, Compiler Internals
Status: Architectural Behavioral Research — These behaviors are rooted in the official language specification. Findings reported to CERT-IL (#11265) and MITRE for documentation.
Executive Summary
During a targeted fuzzing research pipeline on the SBCL compiler (LISP), I focused on the Macro Expansion phase. The goal was to investigate how the compiler enforces lexical isolation boundaries during macro expansion. Using an AI-assisted automation process, I identified three architectural behaviors rooted in the official language specification — behaviors that, in an adversarial context, enable lexical scope isolation bypass. These findings were reported to CERT-IL (#11265) and MITRE for documentation.

1. Problem Definition

The core research question was how the compiler enforces lexical isolation boundaries during macro expansion. Since macros manipulate code before the runtime Stack Frame is allocated, I sought to determine if a compile-time manipulation could cause a variable to leak or change its logical state within a local environment.

2. Methodology: Grammar-Aware Fuzzing

Compilers immediately crash on random input due to Syntax errors, making traditional fuzzing ineffective for finding deep logical flaws. Therefore, a semantically focused automated harness was built:

  • Grammar-Aware Mutation: The fuzzer generated exclusively valid Abstract Syntax Trees (ASTs) using standard grammar to bypass Lexing and Parsing validation successfully.
  • Semantic Oracle: Instead of looking for memory crashes (Segfaults), the oracle monitored semantic anomalies via stdout. A breach was flagged the moment a strictly isolated lexical variable altered its state due to macro execution, without the compiler identifying an error or halting the build.

3. Architectural Findings and Java Generics Analogy

To grasp the nature of this finding in SBCL, it is highly useful to compare it to type-safety models in Java Generics, specifically the PECS principle and the use of ? super T.

In Java, the compiler operates with strict pessimism: any doubt regarding type safety triggers a compilation error. For instance, in a List<? super Apple>, the compiler prohibits inserting a generic parent object like Fruit (via add), because the list might strictly contain Apples. This strict enforcement aims to prevent memory pollution and runtime crashes.

In SBCL, the paradigm is reversed. The compiler does not validate operations against the active Lexical Environment (LEXENV) during macro expansion. This oversight allowed me to isolate three security-relevant architectural behaviors, which are rooted directly in the official language specification:

Vector 1: Dynamic Binding Injection

This vector exploits a discrepancy in the compiler's Pass Order, backed by the SPECIAL declaration defined in the ANSI Specification [1]. The following flowchart illustrates the compilation pipeline and the exact point of code injection:

[Step 1: Front-End] Read & Parse (AST Generation)
                      |
                      v
[Step 2: Lexical Analysis] Register "secret-token" as LOCAL in LEXENV
                      |
                      v
[Step 3: Macro Expansion] -> Runs inject-dynamic-binding macro
                           * Evaluates: (proclaim '(special secret-token))
                           * Global Symbol Table is OVERRIDDEN immediately!
                      |
                      v
[Step 4: Code Generation] Compiler checks Symbol Table for "secret-token"
                           * Sees dynamic SPECIAL_FLAG set in Step 3
                           * CRITICAL BYPASS: Generates Dynamic Runtime Lookup!
                      |
                      v
[Step 5: Back-End] Final Assembly/Bytecode generation (Isolation broken)

What is actually happening here?
In Step 2, the compiler registers the variable secret-token as isolated (Lexical) within the let block. However, in Step 3, during macro expansion, the proclaim directive modifies the symbol's metadata globally, defining it as SPECIAL (dynamic). As documented in CLtL2 [2], this declaration forces all subsequent bindings to act dynamically.

When the compiler reaches Step 4 (Code Generation), it checks the symbol table, recognizes the newly implanted global flag, and generates a Dynamic Runtime Lookup instruction rather than a secure lexical reference. The local scope boundary is eradicated at the binary level, allowing external code to access the variable.

;; PoC Code: dynamic_binding_attack.lisp
(defmacro inject-dynamic-binding (var-name)
  (proclaim `(special ,var-name)) ; Immediate mutation of global symbol metadata inside compilation phase
  `(inline-log "Flag compilation modified"))

(defun target-function ()
  (let ((secret-token "ENCRYPTED_DATA")) ; Supposed to be immutable and isolated lexically
    (inject-dynamic-binding secret-token) ; Triggering macro expansion and flag override
    secret-token)) ; Compiler now generates dynamic runtime lookup instead of safe lexical offset

;; Under vulnerable compilers, external call stack manipulation can now hijack 'secret-token'

Vector 2: Global Symbol Table Pollution

In LISP, a Symbol is a physical singleton object in memory [4]. The intern function locates or creates symbols in the global Symbol Table. According to the HyperSpec [3], if a symbol with the same name is already accessible, intern returns the existing pointer. When a macro executes this function on an existing lexical variable name (like internal-key), it secures a direct, public, global pointer to it.

This action establishes an "Overlapping Reference": a single memory cell with two pointers—one internally protected, and one externally controlled by the macro. The macro abuses this external pointer to implant data onto the object's Property List via the setf function. Since symbol properties are globally exposed and explicitly defined as destructive in the specification [4], an attacker can utilize this to exfiltrate data from heavily shielded scopes, making it accessible to external code.

;; PoC Code: symbol_interning_attack.lisp
(defmacro pollute-symbol-table (sym-name)
  (let ((global-sym (intern (string sym-name))))
    (setf (get global-sym 'compromised-leak) t)
    `(quote ,global-sym)))

(defun secure-enclave ()
  (let ((internal-key "PRIVATE_KEY"))
    (pollute-symbol-table internal-key) ; Context confusion during compilation
    ;; Execution trace leaks internal-key bounds to global symbol property listings
    ))

Vector 3: Cross-Package Namespace Confusion

Unlike Vector 1 which alters the symbol's Type, Vector 3 corrupts its Routing across namespaces. According to Chapter 11 of the ANSI Spec [5], the shadow function alters system consistency rules, hiding an existing symbol and generating a new symbol under the same name.

By dynamically spawning a package at compile-time (via make-package) and calling shadow on the local variable's name, the macro obliterates the compiler's lexical tracking table. When the compiler subsequently attempts to resolve the address of the local variable auth-token, the hijacked routing deceives it, redirecting execution to the global public namespace (cl-user) rather than the protected local stack.

;; PoC Code: package_hijack_attack.lisp
(defmacro hijack-namespace (target-sym)
  (let ((shadow-pkg (make-package "EPHEMERAL-ATTACK-PKG" :use '("CL"))))
    (shadow (string target-sym) shadow-pkg) ; Hijacking compilation target pointer
    `(inline-log "Namespace tracing broken")))

(defun sensitive-scope ()
  (let ((auth-token "SESSION_VALID"))
    (hijack-namespace auth-token) ; Compilation context dereferences token from local frame
    auth-token))

4. Threat Model: Supply Chain Attacks

Architectural behaviors of this nature pose a critical threat to CI/CD pipelines. An attacker can publish a package containing a "Poisoned Macro". When a build server compiles the project, the macro triggers, breaks out of its scoping boundary, and silently accesses sensitive variables (such as API keys or encryption secrets) belonging to other modules in the environment, all without triggering runtime alerts.
